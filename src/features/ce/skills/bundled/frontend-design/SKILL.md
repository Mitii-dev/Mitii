# Frontend Design

## Quick Reference

- Start with the product, audience, and job of the screen; make the first viewport specific to that subject.
- Choose a restrained but distinctive palette, typography system, layout rhythm, and one memorable visual signature.
- Avoid generic AI defaults, decorative clutter, and visible instructional copy about how the UI works.
- Build the actual usable screen or app first, not a marketing shell unless the user asked for a landing page.
- Verify screenshots or rendered output on mobile and desktop when a runnable frontend exists.

## Scope

Use this skill when the user asks to design, redesign, polish, restyle, or visually implement frontend pages, product screens, dashboards, landing pages, portfolios, games, or interactive tools.

Do not use it for backend tasks, documentation-only work, minor copy edits, simple bugfixes, exact Figma implementation, or reusable component API design unless the visual direction is the main work.

## Procedure

1. Identify the subject, audience, primary user job, and visual constraints from the request and existing app.
2. Inspect existing design system files, routes, components, CSS, assets, and screenshots if present.
3. Form a compact design direction: 4-6 colors, 2-3 type roles, layout concept, interaction tone, and one signature element.
4. Build with domain-specific content and real app affordances. Avoid filler sections, nested cards, one-note palettes, and unsupported decorative effects.
5. Ensure responsive stability: no overlapping text, stable dimensions for fixed-format controls, keyboard focus, reduced-motion handling, and readable contrast.
6. Verify by running the app or static preview when available, then inspect desktop and mobile rendering with screenshots or equivalent checks.

## Design Rules

- Hero-scale type belongs only in true hero contexts.
- Cards are for repeated items, modals, and framed tools; page sections should be unframed layouts or full-width bands.
- Use icons for compact tool controls when the app already has an icon library.
- Use real or generated bitmap imagery for websites where visual assets are expected.
- Do not use viewport-width font scaling, negative letter spacing, decorative orbs, or generic gradient-only hero art.
- If an existing design system is present, extend it rather than inventing a conflicting visual language.

## Output

For implementation tasks, summarize the visual direction, files changed, and rendering checks. For advice-only tasks, provide the direction and concrete implementation notes without editing files.

## Completion

Finish when the screen matches the intended visual direction, works at relevant breakpoints, preserves usability/accessibility, and has been visually checked or the unavailable check is reported.
